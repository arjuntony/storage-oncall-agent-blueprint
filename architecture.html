How It Works — Claude Code vs. This Agent

Claude Code (what you know)

You type →
Claude CLI →
1 API call to Claude →
Response

Single request, single response.
Claude sees your code directly.

Storage Agent (what we're building)

You type →
Streamlit UI →
LangGraph → LOOP:
  API call #1 → tool call → run YOUR code
  API call #2 → tool call → run YOUR code
  API call #3 → final answer
Response

Multiple API calls in a loop.
Claude sees tool RESULTS, not your infra directly.

Architecture — Request Flow

👤 You (Browser / CLI)

Type: "Triage MDS switch lva1-mds01 — ports flapping"

Streamlit :8501

🖥️ Frontend (Streamlit)

Chat interface receives your message, passes to agent engine

Python Send to agent

🧠 Agent Engine (LangGraph ReAct)

The brain. Takes system prompt + your message + tool definitions.
Calls Claude API in a loop — Claude decides which tools to call,
reads results, calls more tools, until it has a final answer.

LangGraph LangChain Tool Loop

🔧 Your Tools (Python Functions)

mds_port_status(), array_health(), esxi_vm_status(), etc.
POC: return synthetic data. Later: call real APIs (MDS NX-API, vCenter, etc.)

Python @tool Execute locally

☁️ Claude API ●

POST https://api.anthropic.com/v1/messages
The ONLY external dependency. Same API as Claude Code uses.
Each loop iteration = 1 API call. Typically 3-6 calls per triage.

Anthropic SDK HTTPS outbound

ReAct Loop — Live Triage Example

User says: "Triage incident STOR-2001 — storage array degraded"

C1

Claude thinks: "I need to see the incident first"

Calls: get_storage_incidents(status="active")

T1

Tool returns: 3 active incidents

STOR-2001: array degraded, 2 disk failures, 47 VMs affected

C2

Claude thinks: "Disk failures — check the array health"

Calls: array_health(array_name="stor-lva1-array05")

T2

Tool returns: Array degraded

RG02 lost redundancy (disk-2-14 failed), RG03 rebuilding (43.2%), write latency 8.5ms

C3

Claude thinks: "Need the disk failure runbook + VM impact"

Calls: search_runbooks("disk failure") + esxi_vm_status("esxi-lva1-host10")

T3

Tools return: Runbook found + ESXi status

disk-failure.md matched, 23 VMs on host, memory 92%, datastore latency 45ms

C4

Claude thinks: "Load the runbook for remediation steps"

Calls: load_runbook("disk-failure.md")

T4

Tool returns: Full runbook text

Triage steps, remediation procedures, escalation paths

F

Claude synthesizes final triage report

Root cause + blast radius (47 VMs) + immediate actions + escalation. No more tool calls — done.

5 API calls to Claude (C1-C4 + final) • 6 tool executions (all local Python) • ~15 seconds total

What the Claude API Call Actually Looks Like

This is what LangGraph sends on iteration 2 (after the first tool result comes back):

POST https://api.anthropic.com/v1/messages

{
  "model": "claude-sonnet-4-5-20250929",
  "system": "You are a Storage Oncall Assistant...",
  "messages": [
    {"role": "user", "content": "Triage STOR-2001"},
    // Claude's previous response — it chose to call a tool
    {"role": "assistant", "content": [{
      "type": "tool_use",
      "name": "get_storage_incidents",
      "input": {"status": "active"}
    }]},
    // Your tool's output — injected by LangGraph
    {"role": "user", "content": [{
      "type": "tool_result",
      "content": "{\"incidents\": [{\"id\": \"STOR-2001\", ...}]}"
    }]},
  ],
  "tools": [
    {"name": "mds_port_status", "description": "Get MDS switch...", "input_schema": {...}},
    {"name": "array_health", "description": "Get storage array...", "input_schema": {...}},
    // ... all 11 tools listed here
  ]
}

Key insight: Claude sees tool definitions (name + description + schema) and tool results.
It never sees your infrastructure directly. Your Python functions run locally and return data to Claude.

Available Tools

🚨 get_storage_incidents

Active incidents with severity, alerts, affected hosts/VMs. Always call FIRST.

🔌 mds_port_status

MDS switch port state — flap count, CRC errors, speed, connected devices.

🏷️ mds_zoneset

Active FC zoneset — zone names, WWPN members, login status.

🖥️ esxi_vm_status

ESXi host — CPU, memory, VMs, datastore latency, HBA status.

💿 datastore_health

Datastore detail — capacity, latency, SCSI errors, snapshots, connected hosts.

🗄️ array_health

Storage array — disk state, RAID groups, rebuild progress, I/O performance.

💥 disk_failures

Failed disks — SMART data, replacement tickets, failure rate trend.

🔧 rv_tool_check

RV tool checks — connectivity, firmware, zoning validation.

🌐 fabric_topology

FC fabric — switches, ISL links, path redundancy, connected devices.

🔍 search_runbooks

Search oncall runbooks by keyword — returns matching titles and snippets.

📖 load_runbook

Load full runbook content — triage steps, remediation, escalation.

Graduation Path — POC to Production

Phase 1: POC (Now)

Synthetic data, 11 tools, 5 runbooks. No real APIs needed. Just an Anthropic API key.

1-2 days

Phase 2: Real Incident Data

Connect to your alerting system (PagerDuty, OpsGenie, custom). Replace get_storage_incidents synthetic data with real API calls.

2-3 days

Phase 3: Real MDS + vCenter APIs

Replace synthetic MDS data with Cisco NX-API (JSON-RPC over HTTP). Replace ESXi data with vSphere REST API.

1-2 weeks

Phase 4: Storage Array + RV Tool APIs

Connect to vendor-specific array APIs (Dell PowerStore, NetApp ONTAP, Pure Storage). Integrate RV tool.

1-2 weeks

Phase 5: Smart Routing + Context Management

Add middleware to classify queries (triage vs analytics), filter tools, switch models. Add scratchpad for large outputs.

3-5 days

Phase 6: Slack Bot + CLI

Add Slack integration for oncall notifications. Build CLI for terminal-based triage.

1 week
